## 1. What Is This?

**BHM (Bulk Health Metrics)** is a custom plugin for **Oracle Field Service (OFS)**. Field technicians open it from within an OFS activity to record battery maintenance data for all assets belonging to that activity's customer.

**What a technician does:**
1. Opens the BHM plugin from an OFS activity screen.
2. Sees a list of all installed-base (IB) assets for that customer — pulled live from Oracle Fusion.
3. Clicks any asset row to expand a detail card.
4. Fills in: checklist items, electrical readings (Low VDC, High VDC, Low SG), component condition ratings, and a problem note.
5. Saves the record (stored locally in the browser). Data is later submitted to Oracle Integration Cloud (OIC).

## 4. How an OFS Plugin Works

OFS plugins are web applications loaded inside an `<iframe>` inside the OFS technician portal. They communicate with OFS using `window.postMessage`.

**Lifecycle:**

```
Plugin sends:  { method: 'ready', sendInitData: true }
OFS replies:   { method: 'init', attributeDescription: {...} }
Plugin sends:  { method: 'initEnd' }
OFS replies:   { method: 'open', openParams: {...}, activity: {...} }
                         ↑ this is when the plugin gets its data
Plugin sends:  { method: 'close' }   ← when done
```

- The `init` message contains metadata about OFS property definitions (e.g. enum values for `problem_code`). This is saved to localStorage so it survives page reloads.
- The `open` message contains the **live activity data** — the customer name, activity ID, and any OFS properties configured for this plugin.

This entire handshake is managed by [ofsc-plugin-api-transport.js](#ofsc-plugin-api-transportjs).

## 5. Authentication — Two Token Types

BHM uses **two separate OAuth application keys** registered in OFS:

| Constant | Key string | OFS application name | Purpose |
|----------|-----------|---------------------|---------|
| `Constants.KEY_EXT` | `'Ext'` | Concentric - Oracle Fusion Applications - User Asserted | OAuth Bearer token for calling Oracle Fusion REST APIs (IB Assets, etc.) |
| `Constants.KEY_INT` | `'ofsApiApplication'` | App CX Service | Token for internal Field Service APIs |

**How token acquisition works:**
1. Code calls `ofscConnector.obtainToken(Constants.KEY_EXT)`.
2. This sends a `callProcedure / getAccessToken` postMessage to OFS.
3. OFS performs the OAuth flow and returns the token in `data.resultData`.
4. The token is added as `Authorization: Bearer <token>` on each Fusion HTTP request.

> Tokens are fetched **fresh on every request** — BHM does not cache them.

## 6. Application Startup Flow

```
index.html loads
   ↓
JET bootstraps → loads asset.js (the root ViewModel)
   ↓
AssetViewModel constructor()
   └─ calls _loadProblemCodes()
         └─ waits for app.loaded (OFS handshake to complete)
               ↓
         _processOpenData(openData, attrDesc)
            ├─ Extracts problem_code enum (4 fallback methods, see §8)
            ├─ Sets window.__problemCodeEnum, window.__activityId
            └─ Reads wo_account_name from openData
                  ↓ (if account name found)
            IbAssetsService.getAssetsByAccountName(accountName)
                  └─ GET Fusion IB Assets API → maps response
                        ↓
                  window.__ibAssets = [ mapped assets... ]
```

Then when the technician taps **"View Assets"**:

```
navigateToAllAssets()
   ├─ Waits for __ibAssetsReady Promise
   ├─ Dynamically injects asset-detail.js + all-asset-details.js scripts
   └─ Calls AllAssetDetails.render(containerId, data)
         where data = window.__ibAssets (array) OR this._enumValues (object fallback)
```

## 7. Data Flow — From OFS to Screen

```
Oracle Field Service
  │
  │  postMessage (open event)
  │  openParams.properties.wo_account_name = "Campbell Soup Co."
  │  activity.aid = "300000005911718"
  ▼
ofsc-plugin-api-transport.js
  │  saves attributeDescription to localStorage
  │  resolves app.loaded promise
  ▼
asset.js  (_processOpenData)
  │  extracts wo_account_name
  │  sets window.__activityId
  ▼
ib-assets-service.js
  │  GET /fscmRestApi/resources/11.13.18.05/installedBaseAssets
  │      ?q=CustomerSitePartyName=Campbell Soup Co.
  │  Authorization: Bearer <token from KEY_EXT>
  ▼
fusion-rest-api-transport.js  (makes the actual HTTP fetch)
  │
  │  response.items = [ { AssetId, SerialNumber, Description, ItemNumber, ... }, ... ]
  │
  ▼
IbAssetsService._mapItem()
  │  maps each raw Fusion item to BHM's internal shape:
  │  { instanceNumber, assetId, serial, mfr, model, desc, mfgDate, barcode, isBat, isFlood }
  ▼
window.__ibAssets = [ mapped assets... ]
  ▼

  │  detects Array → IB mode
  │  renders search bar + filter tabs + table header
  │  for each asset: calls AssetDetail.createRows(asset, idx)
  ▼
AssetDetail.createRows()
  │  builds summary <tr> (collapsed row with asset info)
  │  builds detail  <tr> (expandable card with all form fields)
  └─ returns DocumentFragment appended to <tbody>
```

## 8. Key Source Files Explained

### constants.js
**Path:** [src/js/constants.js](src/js/constants.js)

A single AMD module that exports a `Constants` class with only static getters. Think of it as a central config dictionary.

Key constants:
```js
Constants.KEY_EXT          // 'Ext' — application key for Fusion OAuth
Constants.KEY_INT          // 'ofsApiApplication' — application key for OFS internal APIs
Constants.FUSION_BASE_URL  // 'https://ibfaqy-dev1.fa.ocs.oraclecloud.com:443'
Constants.ERR_AUTHENTICATION  // user-facing error string
Constants.ERR_SERVER          // user-facing error string
```

---

### required-properties.json
**Path:** [src/js/required-properties.json](src/js/required-properties.json)

OFS reads this file when the plugin is uploaded. It declares:
- `plugin.label` — the system identifier in OFS (e.g. `bulk_health_metrics`)
- `plugin.val` — the display name shown in the OFS portal (`Bulk Health Metrics`)
- `pluginAppsObj` — the two OAuth application keys (Ext and Int) that OFS must have configured for this plugin
- `properties` — any OFS properties this plugin requires (currently none listed here; `problem_code` is optional)

> **Note on the OFS label:** Once a plugin is registered in OFS, its `label` field is locked and cannot be changed through the OFS UI. Only the `val` (display name) can be updated. The value `bulk_health_metrics` in this file is what was specified at registration time.

---

### ofsc-connector.js
**Path:** [src/js/ofsc-connector.js](src/js/ofsc-connector.js)

The low-level messaging bridge. It wraps `window.postMessage` in a Promise-based API.

Key methods:
- `sendMessage(data)` → sends a message to the OFS parent frame, returns a Promise that resolves when OFS replies
- `obtainToken(key)` → sends `callProcedure / getAccessToken` to OFS, returns a Promise resolving to `{ token: '...' }`
- `onPostMessage(event)` → listener that receives messages from OFS and routes them to the correct Promise callback

The connector handles **one non-procedure message at a time** (channel busy protection). Procedure calls (like token requests) get a unique `callId` so they can be tracked concurrently.

---

### ofsc-plugin-api-transport.js
**Path:** [src/js/services/ofsc-plugin-api-transport.js](src/js/services/ofsc-plugin-api-transport.js)

Sits on top of `OfscConnector` and manages the plugin lifecycle:

1. `load()` sends `{ method: 'ready', sendInitData: true }`.
2. Handles the `init` reply → saves `attributeDescription` to localStorage → sends `initEnd`.
3. Handles the `open` reply → validates required properties → resolves the `app.loaded` Promise that unblocks the rest of the app.

`_verifyProperties()` checks that all properties listed in `required-properties.json` are present in `attributeDescription`. If any are missing, it throws `ApplicationCriticalError` and the app shows an error screen.

---

### fusion-rest-api-transport.js
**Path:** [src/js/services/fusion-rest-api-transport.js](src/js/services/fusion-rest-api-transport.js)

A reusable, authenticated HTTP client for Oracle Fusion REST APIs. It knows nothing about IB Assets specifically — it's a generic layer.

**Usage pattern:**
```js
const transport = new FusionRestApiTransport(Constants.FUSION_BASE_URL, ofscConnector);
const data = await transport.request(
    '/some/fusion/path',
    FusionRestApiTransport.HTTP_METHOD_GET,
    { q: 'SomeField=SomeValue' }   // optional query params
);
```

**What it does internally:**
1. Calls `ofscConnector.obtainToken(Constants.KEY_EXT)` to get a Bearer token.
2. Builds the full URL from `baseUrl + path + ?queryString`.
3. Adds `Authorization: Bearer <token>`, `Content-Type: application/json`, `Accept: application/json`.
4. Calls `fetch()` and returns the parsed JSON.

**Important quirk — URL encoding:**
Fusion's REST API uses `q=Field=Value` syntax (an equals sign inside the query param value). Standard `URLSearchParams` would encode the inner `=` as `%3D`, breaking the filter. The transport manually encodes values with `encodeURIComponent()` then restores literal `=` signs.

---

### ib-assets-service.js
**Path:** [src/js/services/ib-assets-service.js](src/js/services/ib-assets-service.js)

BHM-specific service that fetches installed-base assets from Fusion for a given customer.

**API endpoint:**
```
GET /fscmRestApi/resources/11.13.18.05/installedBaseAssets
    ?q=CustomerSitePartyName=<wo_account_name>
```

**`_mapItem(item)` — field mapping:**

| BHM field | Fusion API field | Notes |
|-----------|-----------------|-------|
| `instanceNumber` | `item.AssetId` | Used as unique key for localStorage and row ID |
| `assetId` | `item.AssetId` | Displayed in the Asset ID column |
| `serial` | `item.SerialNumber` | Displayed in Serial column |
| `desc` | `item.ItemNumber` | Displayed as the description pill (e.g. `TEST_BATTERY`) |
| `mfr` | *(not available yet)* | Left blank — Fusion API doesn't expose this field currently |
| `model` | *(not available yet)* | Left blank |
| `isBat` | derived from `item.Description` | `true` if Description matches `/battery|bat\b/i` |
| `isFlood` | derived from `item.Description` | `true` if Description matches `/flooded|lead.?acid/i` |

`isBat` controls whether the expanded card shows the **Components section** (battery-specific) and whether **Low SG** reading is shown. Battery assets (BAT-) show 2 readings; non-battery assets show 3.

### asset.js (ViewModel)
**Path:** [src/js/viewModels/asset.js](src/js/viewModels/asset.js)

The root Knockout ViewModel for the landing screen. This is where everything starts.

**Observables:**
- `showAllAssets` — controls which view is visible (landing card vs. asset table)
- `problemCodeOptions` — array of `{ index, label }` objects (for the legacy enum fallback)
- `problemCodeCount` — computed count shown in the header badge

**`_processOpenData(openData, attrDesc)`** — the main init logic:
1. Tries 4 methods to extract `problem_code` enum values (for fallback display).
2. Sets `window.__problemCodeEnum`, `window.__activityId`, `window.__ibAssets = null`.
3. Reads `wo_account_name` from `openParams.properties` or `openData.activity`.
4. If account name found: calls `IbAssetsService.getAssetsByAccountName(accountName)` and stores the result Promise in `this._ibAssetsReady`.

**`navigateToAllAssets()`:**
1. Sets `showAllAssets(true)` to show the list view.
2. Awaits `_ibAssetsReady` (so we don't render before the API call finishes).
3. Dynamically loads `asset-detail.js` and `all-asset-details.js` as `<script>` tags (they are plain-JS IIFE modules, not AMD).
4. Calls `AllAssetDetails.render(containerId, data)` where `data` = `window.__ibAssets` if available, else the enum object.

---

### all-asset-details.js
**Path:** [src/js/viewModels/all-asset-details.js](src/js/viewModels/all-asset-details.js)

A plain-JS IIFE (Immediately Invoked Function Expression) — **not an AMD module**. It exposes a global `AllAssetDetails` object.

**`render(containerId, data)`:**
- If `data` is an **Array** → IB Assets mode (live from Fusion)
- If `data` is an **Object** → legacy `problem_code` enum mode (fallback)
- Renders: search input + filter tabs (All / Pending / Completed) + an HTML table
- Each row is created by `AssetDetail.createRows(asset, idx)` or `AssetDetail.createRows(key, label, idx)`

**`applyFilter(containerId)`:** Called on every keystroke in the search box. Filters visible rows by:
- Search text matching `assetId`, `serial`, `label`, or `code` (stored as `data-*` attributes on the `<tr>`)
- Active filter tab: All / Pending / Completed (reads `Complete` checkbox state + localStorage fallback)

**`setFilter(containerId, filterVal, btn)`:** Activates a filter tab and triggers `applyFilter`.

---

### asset-detail.js
**Path:** [src/js/viewModels/asset-detail.js](src/js/viewModels/asset-detail.js)

Another plain-JS IIFE, exposing the global `AssetDetail` object. This is the most complex file.

**`createRows(codeIndexOrAsset, codeLabelOrRowIndex, rowIndex)`**

Accepts two calling signatures:
```js
// IB Asset mode (from Fusion):
AssetDetail.createRows(assetObject, rowIndex)

// Legacy enum mode (fallback):
AssetDetail.createRows(codeIndex, codeLabel, rowIndex)
```

Returns a `DocumentFragment` with two `<tr>` elements:
1. **Summary row** (`cc-sum-<ri>`) — always visible: expand button, view button, description pill, serial, manufacturer, model, status badge
2. **Detail row** (`cc-det-<ri>`) — hidden by default, shown when expanded: the full data card

**`_buildPanel(d, label, code, ri)`** — generates the HTML for the expanded card. The card has 4 horizontal sections:

| Section | CSS class | Contents |
|---------|-----------|----------|
| Asset Info | `cc-sec-asset` | Asset ID, Serial, Manufacturer, Model, Description, Mfg Date, Age, Barcode; Last Completed dates; Maintenance checklist |
| Last Reading | `cc-sec-reading` | Last Complete date box; Low VDC, High VDC, Low SG inputs |
| Components | `cc-sec-comps` | Per-component condition dropdowns (1–5 scale) — BAT assets only |
| Problem Note | `cc-sec-note` | Auto-populated textarea; Needs Repair checkbox; Complete checkbox |

**`toggle(ri)`** — expands or collapses a row. On open: calls `_restoreRecord(ri)` to load saved data from localStorage, then `_initCompactSelects(ri)` to set up the compact dropdown behavior.

**`updateNote(ri)`** — called via `oninput` on every checklist checkbox and reading input. Auto-populates the Problem Note textarea by combining:
- Checked checklist items → `"Checklist: Visual Inspection, Added Water"`
- Non-default component ratings → `"Components: Cables: 3 - Safety Related Issue"`
- Reading values → `"Readings: Low VDC: 7.7, High VDC: 6"`

**`saveRow(ri)` / `continueRow(ri)`** — save to localStorage. `continueRow` also collapses the row after saving.

---

### persistent-storage.js
**Path:** [src/js/storage/persistent-storage.js](src/js/storage/persistent-storage.js)

A minimal wrapper around `window.localStorage`:
```js
PersistentStorage.saveData(key, value)   // JSON.stringify → setItem
PersistentStorage.loadData(key)          // getItem → JSON.parse (returns {} if missing)
PersistentStorage.removeData(key)        // removeItem
```

Used by `ofsc-plugin-api-transport.js` to persist `attributeDescription` across sessions. The asset form data is saved directly in `asset-detail.js` using raw `localStorage` calls (not this wrapper).

## 10. Offline / LocalStorage Persistence

All form data is saved to `localStorage` when the technician clicks **Save** or **Continue**.

**Storage key format:**
```
mae_record_<activityId>_<codeIndex>
```
- `activityId` = `window.__activityId` (from `openData.activity.aid`)
- `codeIndex` = `asset.instanceNumber` (IB mode) or the problem_code key (legacy mode)

**Saved record shape:**
```js
{
  rowIndex:    0,
  note:        "Checklist: Visual Inspection\nReadings: Low VDC: 7.7",
  checklist:   { "Visual Inspection": true, "Added Water": false, ... },
  components:  { "Cables": "1", "Connectors": "3", ... },
  readings:    { "Low VDC": "7.7", "High VDC": "6" },
  needsRepair: false,
  complete:    true,
  savedAt:     "2025-03-12T10:30:00.000Z"
}
```

When a row is **expanded**, `_restoreRecord(ri)` reads this record and populates all form fields automatically.

The status badge (Saved / Complete) is refreshed by `_refreshSummaryStatus(ri)` immediately after render (via `setTimeout(fn, 0)` to let the DOM settle) and after every save.

## 11. Global Variables (window.__)

These are set by `asset.js` and read by the plain-JS scripts (`all-asset-details.js`, `asset-detail.js`) that can't use AMD `define([...])` imports:

| Variable | Set by | Read by | Value |
|----------|--------|---------|-------|
| `window.__activityId` | `asset.js` | `asset-detail.js` | OFS activity ID string (used in localStorage key) |
| `window.__problemCodeEnum` | `asset.js` | `all-asset-details.js` | `{ codeIndex: codeLabel }` object — legacy fallback |
| `window.__ibAssets` | `asset.js` (via `ib-assets-service.js`) | `asset.js`, `all-asset-details.js` | Array of mapped IB asset objects, or `null` if unavailable |

**Fallback logic:** If `window.__ibAssets` is `null` (network error, offline, or `wo_account_name` missing), `all-asset-details.js` falls back to rendering rows from `window.__problemCodeEnum`.

## 12. OFS Plugin Registration

To register BHM in an OFS instance:

1. **Build** the plugin: `grunt` → produces `build/mae-<version>-hosted.zip`
2. In OFS: **Configuration → Forms & Plugins** → add new plugin
   - Type: Plugin Archive
   - Upload the zip file
   - Label (auto-set from `required-properties.json`): `bulk_health_metrics`
   - Name (editable): `Bulk Health Metrics`
3. **Applications tab** — add both OAuth app entries:
   - `Ext` → type `oauth_user_assertion` → key `fusionOAuthUserAssertionApplication`
   - `Int` → type `ofs` → key `ofsApiApplication`
4. **Screen configuration** — add BHM button to the "Edit/View activity" screen for the relevant user types.

> **Important:** The `label` field in OFS is locked after initial registration. Do not attempt to change it. The `val` (display name) can be freely updated.

## 14. Debugging Tips

**Browser DevTools — Console:**

When the plugin loads, it logs extensively:
```
[AssetViewModel] openData received: {...}
[DEBUG] openData keys: [...]
[DEBUG] openData.activity: {...}
[AssetViewModel] wo_account_name: "Campbell Soup Co."
[AssetViewModel] IB Assets loaded: 5
[AllAssetDetails] Rendered 5 row(s) (IB Assets)
```

If you see `wo_account_name not found` → the OFS activity either doesn't have this property configured, or it's empty for this customer.

**Network tab:**
- Look for a request to `ibfaqy-dev1.fa.ocs.oraclecloud.com` with path `/fscmRestApi/resources/11.13.18.05/installedBaseAssets`
- It should have `Authorization: Bearer <long token string>`
- Response should be `{ items: [...], count: N }`

**localStorage (Application tab → Local Storage):**
- `plugin_attributeDescription` — cached OFS property metadata
- `mae_record_<activityId>_<instanceNumber>` — saved form records per asset

**Common issues:**

| Symptom | Likely cause |
